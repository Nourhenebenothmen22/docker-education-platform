1. Qu'est-ce que Docker ?

Docker est une plateforme open-source qui automatise le déploiement d'applications dans des conteneurs logiciels légers et portables. Il permet aux développeurs d'empaqueter une application avec toutes ses dépendances (bibliothèques, runtime, variables système) dans une seule unité logique appelée image.

Quel problème résout Docker ? : Le problème principal résolu par Docker est le fameux syndrome du « Ça fonctionne sur ma machine » (It works on my machine). Avant Docker, déplacer une application d'un ordinateur de développement vers un serveur de production provoquait souvent des erreurs à cause de différences de versions d'OS, de bibliothèques ou de configurations système. Docker garantit que l'application s'exécutera exactement de la même manière partout.

Concepts Clés de Docker

Conteneur

Une instance active et isolée d'une image Docker. C'est l'équivalent d'un processus système encapsulé qui s'exécute de façon autonome.

Image

Un modèle en lecture seule contenant le code, le runtime, les dépendances et la configuration nécessaires pour lancer un conteneur. C'est le plan de construction.

Docker Engine

Le cœur de Docker. C'est l'application client-serveur (composée du daemon dockerd et de la CLI) installée sur la machine hôte pour créer et gérer les conteneurs.

Docker Hub

Le registre officiel hébergé sur le cloud où l'on peut partager, stocker et télécharger des images Docker préconfigurées (comme Node.js, Python, Nginx, PostgreSQL).

Dockerfile

Un simple fichier texte contenant une liste d'instructions séquentielles exécutées par Docker pour construire automatiquement une image personnalisée.

Docker Compose

Un outil pour définir et lancer des applications multi-conteneurs (par exemple, une API Node.js et sa base de données MongoDB) à l'aide d'un seul fichier de configuration YAML.

Différence entre Image et Conteneur

Pour faire une analogie avec la programmation orientée objet : l'image est la Classe (le concept en lecture seule), tandis que le conteneur est l'Objet (l'instance vivante et modifiable en mémoire). On peut lancer des dizaines de conteneurs identiques à partir d'une seule et même image.

Docker vs Déploiement Traditionnel

Dans un déploiement traditionnel, vous devez installer manuellement chaque runtime (ex: Java, Python) sur le système d'exploitation du serveur de production, avec le risque que des applications différentes partagent les mêmes bibliothèques et créent des conflits. Avec Docker, chaque application embarque son propre runtime et ses dépendances de manière isolée.

4. Pourquoi Docker est-il important ?

Environnement de développement cohérent

Avec Docker, l'environnement de développement local est identique à l'environnement de production. Plus de surprises désagréables dues à des configurations différentes entre serveurs.

Résout le problème « Ça marche sur ma machine »

L'application s'exécute dans un conteneur standardisé qui isole toutes les dépendances. Si le conteneur tourne chez le développeur, il tournera exactement de la même manière en production.

Déploiement et mise à l'échelle plus rapides

Le démarrage instantané permet de scaler (mettre à l'échelle) horizontalement les serveurs en quelques secondes. Idéal pour encaisser des hausses de trafic soudaines.

Intégration et Déploiement Continus (CI/CD)

Les pipelines CI/CD construisent des images Docker qui sont testées, poussées vers un registre, puis déployées directement sans aucune configuration manuelle sur les serveurs.

Adapté aux architectures microservices

Chaque microservice (authentification, panier, paiement) peut être placé dans son propre conteneur avec sa propre version de runtime sans interférer avec les autres.

Déploiement cloud et portabilité multi-cloud

Toutes les grandes plateformes cloud (AWS, Azure, GCP) supportent Docker de manière native. Migrer un service d'un fournisseur cloud à un autre est un jeu d'enfant.

6. Guide d'Installation de Docker

Avant de commencer ce lab, vérifiez que Docker est installé avec docker --version.

Sur Ubuntu/Debian, Docker peut être installé via le dépôt officiel Docker. Cette méthode est plus propre qu'une installation manuelle car elle permet de recevoir les mises à jour correctement.

Mettez à jour les paquets et installez les dépendances :

bash
sudo apt-get update
sudo apt-get install -y apt-transport-https ca-certificates curl software-properties-common

Ajoutez la clé GPG de Docker :

bash
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo apt-key add -

Ajoutez le dépôt Docker :

bash
sudo add-apt-repository "deb [arch=amd64] https://download.docker.com/linux/ubuntu focal stable"

Installez Docker :

bash
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io

Vérifiez l'installation :

bash
docker --version
sudo docker run hello-world

Optionnel — exécuter Docker sans sudo :

bash
sudo usermod -aG docker $USER

Après avoir ajouté votre utilisateur au groupe docker, vous devez fermer puis rouvrir la session pour que le changement soit appliqué.

N'exécutez pas docker volume prune ou docker system prune si vous voulez conserver les données de vos conteneurs.

3. Problèmes avant Docker

Avant l'adoption massive de Docker, les développeurs et les équipes DevOps faisaient face à de nombreux problèmes récurrents liés au déploiement et à l'exécution des applications.

Dépendances système conflictuelles

Deux applications sur le même serveur pouvaient nécessiter des versions différentes de la même bibliothèque (ex: Python 2 vs Python 3, Node.js 14 vs Node.js 18). Installer l'une pouvait casser l'autre.

Environnement non reproductible

L'environnement de développement n'était jamais identique à la production. Les différences d'OS, de versions de logiciels et de configuration causaient des bugs impossibles à reproduire localement.

Provisionnement lent

Configurer un nouveau serveur prenait des heures, voire des jours. Chaque étape (installation d'OS, dépendances, configuration réseau) devait être faite manuellement ou avec des scripts complexes.

"Ça marche sur ma machine"

Le problème le plus célèbre. Une application fonctionnait chez le développeur mais plantait en production à cause de différences d'environnement. Docker a été créé spécifiquement pour résoudre ce problème.

Docker a résolu tous ces problèmes en standardisant l'environnement d'exécution. Chaque application embarque son propre OS, ses dépendances et sa configuration dans une image portable.

2. Histoire et Apparition de Docker

Docker a été créé par Solomon Hykes et lancé en open-source en mars 2013, à l'époque où il travaillait chez dotCloud (une entreprise de plateforme en tant que service - PaaS). Le projet a immédiatement suscité un engouement massif au sein de la communauté informatique.

Pourquoi Docker est-il apparu ?

Avant Docker, la conteneurisation existait déjà sur Linux (notamment via LXC - LinuX Containers), mais son utilisation était complexe, nécessitait une expertise système avancée, et n'était pas standardisée. De plus, les machines virtuelles (VM) commençaient à montrer leurs limites en termes de lenteur et de surconsommation de ressources pour les déploiements à grande échelle. Docker est apparu pour démocratiser la conteneurisation en y ajoutant une interface en ligne de commande ultra-simple (CLI), un système de couches d'images réutilisables, et une plateforme centrale de partage (Docker Hub).

L'évolution des VMs vers les conteneurs

Dans les années 2000, la virtualisation par VM a permis de découper un serveur physique en plusieurs serveurs virtuels. Cependant, chaque VM devait exécuter son propre système d'exploitation complet (Guest OS), ce qui consommait beaucoup de RAM et de CPU. Les conteneurs ont introduit une approche révolutionnaire : partager le noyau (kernel) du système d'exploitation hôte tout en isolant complètement les processus applicatifs. Cela a permis d'exécuter des centaines de conteneurs sur une machine là où on ne pouvait faire tourner que quelques VMs.

Pourquoi Docker est devenu si populaire ?

Docker a standardisé l'empaquetage des applications. En fournissant une solution simple pour empaqueter, distribuer et exécuter du code de manière identique sur n'importe quel OS hôte, il est devenu le standard de fait de l'industrie. Son adoption a coïncidé avec l'essor du mouvement DevOps et de l'architecture microservices, pour lesquels la vélocité et l'isolation sont cruciales.

5. Architecture Docker

Docker utilise une architecture client-serveur. Le client Docker (CLI) communique avec le daemon Docker (dockerd) qui construit, exécute et gère les conteneurs.

Docker Client (CLI)

Le client Docker est l'outil en ligne de commande (docker) que vous utilisez dans votre terminal. Il envoie des commandes au daemon Docker via une API REST. Un même client peut communiquer avec plusieurs daemons distants.

Docker Daemon (dockerd)

Le daemon Docker (dockerd) est un processus serveur qui tourne en arrière-plan sur la machine hôte. Il écoute les requêtes API, gère les images, les conteneurs, les réseaux et les volumes. C'est le cœur du moteur Docker.

Docker Engine

Le Docker Engine est l'ensemble composé du daemon, de l'API REST et de la CLI. C'est la couche logicielle qui exécute et orchestre les conteneurs. Il utilise les fonctionnalités du noyau Linux (namespaces, cgroups) pour isoler les processus.

Docker Registry

Un registre Docker est un serveur qui stocke et distribue des images Docker. Docker Hub est le registre public par défaut, mais vous pouvez héberger votre propre registre privé (ex: AWS ECR, GitLab Container Registry, Harbor).

Le flux typique : vous tapez 'docker run nginx' dans le terminal. Le client Docker envoie une requête API au daemon. Le daemon vérifie si l'image nginx est disponible localement, sinon il la télécharge depuis Docker Hub. Ensuite, il crée et démarre un conteneur à partir de cette image.

3. Docker vs Machine Virtuelle

Qu'est-ce qu'une Machine Virtuelle ?

Une machine virtuelle (VM) est une émulation logicielle d'un système informatique physique complet. Elle s'exécute par-dessus un outil appelé Hyperviseur (comme VMware, VirtualBox, Hyper-V) et embarque son propre système d'exploitation complet (l'OS invité), ses pilotes et ses applications. Cela engendre une isolation forte mais très lourde en ressources.

Qu'est-ce qu'un Conteneur Docker ?

Un conteneur Docker est un environnement d'exécution isolé qui partage le noyau (kernel) de la machine hôte. Au lieu de virtualiser le matériel sous-jacent, Docker virtualise le système d'exploitation au niveau des processus. Par conséquent, les conteneurs sont extrêmement légers, démarrent presque instantanément et utilisent une fraction minime des ressources système par rapport à une VM.

Migration avancée d'une VM vers Docker

La migration d'applications depuis des machines virtuelles vers des conteneurs Docker est une étape clé dans la modernisation des infrastructures. Cette section explique pourquoi et comment migrer efficacement.

La migration VM vers Docker transforme radicalement la façon dont vous déployez et gérez vos applications.

Pourquoi migrer ?

Portabilité

Une image Docker fonctionne à l'identique sur n'importe quelle machine (dev, staging, production, cloud).

Rapidité

Les conteneurs démarrent en quelques secondes, contre plusieurs minutes pour une VM complète.

Reproductibilité

L'infrastructure est définie dans du code (Dockerfile, compose.yml). Plus de dérive d'environnement.

Automatisation CI/CD

Les images peuvent être buildées, testées et déployées automatiquement dans un pipeline CI/CD.

Déploiement traditionnel sur VM

Flux de travail manuel typique :

bash
1sudo apt update
2sudo apt install -y nodejs npm mongodb nginx
3npm install
4npm run build
5pm2 start server.js

Problèmes :

  • Configuration manuelle, longue et sujette aux erreurs
  • Dérive de l'environnement entre les serveurs
  • Provisionnement lent (heures vs secondes)
  • Conflits de dépendances entre applications
  • Rollback difficile en cas d'erreur
  • Configuration spécifique au serveur
  • Difficulté de passage à l'échelle
  • Environnement de production différent du développement

Étapes de migration

1

Identifier le runtime

Déterminez le langage et la version nécessaires (Node.js 20, Python 3.11, etc.).

2

Identifier les dépendances

Listez toutes les dépendances système et applicatives (package.json, requirements.txt, etc.).

3

Créer un Dockerfile

Écrivez un Dockerfile qui reproduit l'environnement de la VM de façon automatisée.

4

Construire l'image Docker

Utilisez 'docker build -t my-app .' pour créer l'image.

5

Séparer les services en conteneurs

Au lieu de tout installer sur une seule VM, chaque service (app, base de données, cache) a son propre conteneur.

6

Créer un fichier compose.yml

Définissez l'ensemble des services, réseaux et volumes dans un fichier compose.yml.

7

Ajouter des volumes pour la persistance

Montez des volumes pour les bases de données et autres données qui doivent survivre aux redémarrages.

8

Ajouter des réseaux

Créez un réseau dédié pour que les conteneurs communiquent via leurs noms de service.

9

Ajouter des variables d'environnement

Externalisez la configuration (URL de base de données, clés API) dans des variables d'environnement.

10

Exécuter et vérifier

Lancez 'docker compose up -d --build' et vérifiez que l'application fonctionne.

Images et conteneurs Docker

Image Docker

Une image Docker est un modèle immuable en lecture seule qui contient le code, le runtime, les bibliothèques et la configuration nécessaires pour exécuter une application. Elle est construite à partir d'un Dockerfile et sert de plan pour créer des conteneurs.

Conteneur Docker

Un conteneur Docker est une instance exécutable d'une image. C'est un processus isolé qui partage le noyau de l'hôte mais possède son propre système de fichiers, réseau et espace de processus. On peut créer plusieurs conteneurs à partir d'une même image.

Cycle de vie d'un conteneur

Un conteneur Docker passe par plusieurs états tout au long de sa vie. Comprendre ce cycle est essentiel pour gérer efficacement vos conteneurs.

Created (Créé)

Le conteneur a été créé à partir d'une image mais n'a jamais été démarré. Il existe dans le système de fichiers de Docker mais aucun processus ne tourne.

Running (En cours)

Le conteneur est actif et le processus principal (CMD) est en cours d'exécution. Le conteneur peut recevoir des connexions réseau et exécuter des commandes.

Paused (En pause)

Tous les processus du conteneur sont temporairement suspendus via le mécanisme cgroups freeze. Le conteneeur conserve son état mémoire mais n'utilise pas de CPU.

Exited (Arrêté)

Le processus principal du conteneur s'est terminé (volontairement ou sur erreur). Le conteneur existe toujours mais ne consomme pas de ressources CPU. Il peut être redémarré.

Deleted (Supprimé)

Le conteneur a été supprimé avec 'docker rm'. Toutes ses données non persistées (système de fichiers) sont perdues définitivement.

Commandes associées : docker create, docker start, docker stop, docker pause, docker unpause, docker restart, docker rm.

Ne confondez pas image et conteneur. L'image est le plan (immuable), le conteneur est l'instance (modifiable).

7. Tutoriel : Écrire un Dockerfile

Un Dockerfile est la recette de cuisine de votre conteneur. Il contient une suite d'instructions pour configurer un environnement et y exécuter votre code. Nous allons étudier un exemple concret pour une application Node.js.

InstructionDescription
FROMDéfinit l'image de base (souvent un OS ou un runtime préconfiguré comme Alpine Linux ou Node.js). Chaque Dockerfile doit commencer par FROM.
WORKDIRSpécifie le dossier de travail à l'intérieur du conteneur. Toutes les commandes suivantes s'exécuteront dans ce dossier.
COPYCopie des fichiers ou dossiers de votre machine hôte vers le système de fichiers du conteneur.
RUNExécute des commandes système pendant l'étape de construction (build) de l'image (ex: installer des dépendances, compiler du code).
EXPOSEInforme Docker que le conteneur écoutera sur un port réseau spécifique lors de son exécution (purement documentaire, n'ouvre pas le port de façon effective).
ENVDéfinit des variables d'environnement au sein du conteneur, accessibles par vos scripts.
CMDSpécifie la commande par défaut exécutée lors du lancement du conteneur. Il ne peut y avoir qu'une seule instruction CMD par Dockerfile.

Exemple : Dockerfile pour une Application Node.js

dockerfile
1FROM node:20-alpine
2WORKDIR /app
3COPY package*.json ./
4RUN npm install
5COPY . .
6EXPOSE 3000
7CMD ["npm", "run", "dev"]

Explication ligne par ligne

FROM

Nous partons d'une image officielle contenant Node.js version 20 préinstallé. Le suffixe '-alpine' désigne une distribution Linux ultra-légère (environ 5 Mo) optimisée pour la sécurité et la rapidité.

WORKDIR

Nous créons et nous nous positionnons dans le dossier `/app` à l'intérieur du conteneur. Toutes les commandes suivantes s'y exécuteront.

COPY

Nous copions uniquement les fichiers définissant les dépendances (`package.json` et `package-lock.json`) dans notre dossier de travail actuel (./). Cela permet d'optimiser le cache de Docker.

RUN

Nous lançons l'installation des dépendances dans le conteneur. Comme nous n'avons copié que le `package.json` auparavant, Docker ne réexécutera cette étape de build que si ce fichier change.

COPY

Nous copions tout le reste du code source de notre projet local vers le dossier de travail du conteneur.

EXPOSE

Nous documentons le fait que l'application écoute sur le port 3000.

CMD

Nous spécifions la commande à exécuter pour lancer notre serveur de développement au démarrage du conteneur.

Build context et cache

Le contexte de build est le dossier envoyé au daemon Docker. Chaque instruction crée une couche mise en cache. Docker réutilise les couches tant qu'elles n'ont pas changé. L'ordre des instructions est crucial pour optimiser le cache.

Chaque instruction Dockerfile crée une couche (layer).

Multi-stage build

Le multi-stage build permet d'utiliser plusieurs instructions FROM dans le même Dockerfile. Chaque FROM commence une nouvelle étape. Seuls les artefacts nécessaires sont copiés dans l'étape finale, réduisant considérablement la taille de l'image.

dockerfile
1FROM node:20-alpine AS builder
2WORKDIR /app
3COPY package*.json ./
4RUN npm ci
5COPY . .
6RUN npm run build
7 
8FROM node:20-alpine
9WORKDIR /app
10COPY --from=builder /app/dist ./dist
11RUN npm ci --only=production
12EXPOSE 3000
13CMD ["node", "dist/index.js"]

Utilisez .dockerignore pour réduire le contexte de build et accélérer les constructions.

Ne copiez pas node_modules dans l'image. Utilisez .dockerignore pour l'exclure.

Chaque instruction Dockerfile crée une couche. Combinez les RUN avec && pour réduire le nombre de couches.

Préférez les petites images officielles (alpine, slim) quand c'est possible pour des images plus légères.

8. Tutoriel : Docker Compose

Docker Compose simplifie la vie lorsqu'une application a besoin de plusieurs services (par exemple, une API et une base de données). Il permet de tout orchestrer dans un fichier YAML et de démarrer l'infrastructure complète avec une seule commande.

Pourquoi utiliser Docker Compose ?

Au lieu de taper de longues commandes dans le terminal pour démarrer chaque conteneur, configurer les ponts réseau et lier les volumes de stockage, vous écrivez un fichier `compose.yml`. Cela permet de documenter et d'automatiser l'environnement de développement complet de l'équipe.

Exemple : Application Node.js + MongoDB

yaml
1services:
2  app:
3    build: .
4    ports:
5      - "3000:3000"
6    depends_on:
7      - mongo
8 
9  mongo:
10    image: mongo
11    ports:
12      - "27017:27017"

Explication ligne par ligne

services

Indique le début de la liste des conteneurs qui vont composer notre application.

app

Déclare notre premier service nommé `app` (l'application Node.js).

build

Dit à Docker Compose de construire l'image de ce conteneur en utilisant le `Dockerfile` situé dans le dossier courant (.).

ports

Mappe le port 3000 de la machine hôte vers le port 3000 du conteneur. Cela permet d'accéder à l'application via `http://localhost:3000`.

depends_on

Indique que l'application a besoin du service `mongo` pour fonctionner. Docker Compose démarrera donc d'abord le conteneur de base de données avant d'initialiser le nôtre.

mongo

Déclare notre second service nommé `mongo` (la base de données).

image

Au lieu de construire une image personnalisée, nous téléchargeons directement l'image officielle MongoDB depuis Docker Hub.

ports

Mappe le port par défaut de MongoDB (27017) de la machine hôte vers le conteneur.

Services, réseaux et volumes

Compose permet de définir des réseaux personnalisés pour isoler les services, et des volumes pour persister les données. Chaque service peut avoir sa propre configuration réseau.

yaml
services:
  app:
    networks:
      - backend
  mongo:
    networks:
      - backend

volumes:
  mongo_data:

networks:
  backend:

healthcheck et profiles

healthcheck vérifie que le service est prêt. profiles permet de démarrer sélectivement certains services. depends_on avec condition attend le service avant de démarrer.

yaml
services:
  app:
    depends_on:
      mongo:
        condition: service_started
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000"]
      interval: 30s
      timeout: 10s
      retries: 3

  mongo:
    image: mongo:7
    profiles: ["db", "full"]

depends_on ne garantit pas que la base de données est prête. Utilisez healthcheck pour vérifier la disponibilité.

Utilisez les noms de service pour la communication entre conteneurs. Docker Compose configure automatiquement le DNS.

Compose crée un réseau par défaut si aucun réseau personnalisé n'est défini.

Ne supprimez pas les volumes Docker si vous voulez conserver les données MongoDB.

Volumes Docker et données persistantes

Les conteneurs sont éphémères par nature. Quand un conteneur est supprimé, toutes les données écrites dans son système de fichiers sont perdues. Les volumes Docker permettent de persister les données indépendamment du cycle de vie des conteneurs.

Sans volume, une base de données MongoDB dans un conteneur perd toutes ses données lors de la suppression du conteneur. Avec un volume, les données survivent aux redémarrages et aux suppressions.

Volumes nommés

Gérés par Docker. Créés avec 'docker volume create' ou automatiquement dans compose.yml. Stockés dans /var/lib/docker/volumes/. Recommandés pour la production.

Bind mounts

Montent un dossier de la machine hôte directement dans le conteneur. Idéal pour le développement (rechargement à chaud). Moins portable que les volumes nommés.

tmpfs mounts

Montés en mémoire RAM. Très rapides mais non persistants. Utilisés pour des données sensibles ou temporaires.

bash
1docker volume create mongo_data
2docker run -d -v mongo_data:/data/db mongo:7

Réseaux Docker

Les réseaux Docker permettent aux conteneurs de communiquer entre eux et avec le monde extérieur. Chaque conteneur peut être connecté à un ou plusieurs réseaux.

bridge (pont)

Réseau par défaut. Les conteneurs sur le même bridge peuvent communiquer via leur adresse IP. Idéal pour une application standalone.

host (hôte)

Le conteneur partage directement le réseau de l'hôte. Pas d'isolation réseau. Performances maximales mais moins sécurisé.

overlay (superposition)

Permet à des conteneurs sur différentes machines (cluster Swarm) de communiquer comme s'ils étaient sur le même réseau.

macvlan

Attribue une adresse MAC unique à chaque conteneur, le faisant apparaître comme un appareil physique sur le réseau.

none (aucun)

Le conteneur n'a pas d'accès réseau. Utilisé pour des conteneurs de calcul isolés ou de traitement de données sensibles.

Docker Compose crée un réseau DNS interne qui permet aux conteneurs de se joindre via leur nom de service (ex: 'mongo' au lieu d'une adresse IP). Ce mécanisme est automatique et ne nécessite aucune configuration.

Sécurité Docker

Docker offre plusieurs mécanismes de sécurité pour isoler les conteneurs et protéger l'hôte. Bien configuré, Docker peut être plus sécurisé que des applications déployées traditionnellement.

Principe du moindre privilège

Ne donnez aux conteneurs que les permissions strictement nécessaires. Évitez de lancer les conteneurs en tant que root. Utilisez l'instruction USER dans votre Dockerfile.

Images officielles et signatures

Utilisez toujours des images officielles ou signées depuis Docker Hub. Activez Docker Content Trust (DCT) pour vérifier la signature des images avant de les exécuter.

Scan de vulnérabilités

Utilisez 'docker scout' ou des outils tiers (Trivy, Snyk) pour analyser vos images à la recherche de vulnérabilités connues dans les dépendances système et applicatives.

Réseaux isolés

Placez chaque groupe de services sur son propre réseau Docker. Un conteneur de base de données ne devrait pas être accessible depuis le réseau public.

Limitation des ressources

Utilisez les flags --memory, --cpus, --pids-limit pour empêcher un conteneur de monopoliser les ressources de l'hôte (attaque par déni de service).

Système de fichiers en lecture seule

Montez le système de fichiers du conteneur en lecture seule avec --read-only quand l'application n'a pas besoin d'écrire sur le disque.

Optimisation Docker

Optimiser vos conteneurs Docker permet de réduire les coûts, d'améliorer les performances et de renforcer la sécurité. Voici les meilleures pratiques d'optimisation.

Choisir des images de base légères

Préférez alpine, slim ou distroless aux images standard. Une image node:20 fait 1.2 Go, node:20-slim fait 250 Mo, node:20-alpine fait 130 Mo.

Optimiser le cache Docker

Ordonnez les instructions du Dockerfile du moins changeant au plus changeant. Copiez d'abord package*.json, installez les dépendances, puis copiez le code source.

Utiliser le multi-stage build

Séparez la phase de compilation de la phase d'exécution. L'image finale ne contient que les artefacts nécessaires, pas les outils de build.

Minimiser le nombre de couches

Combinez les commandes RUN avec && et . Chaque instruction crée une couche. Trop de couches augmente la taille de l'image.

Utiliser .dockerignore

Excluez les fichiers inutiles (node_modules, .git, *.log) du contexte de build. Cela accélère le build et réduit la taille de l'image.

Éviter d'installer les dépendances de développement

Utilisez 'npm ci --only=production' ou équivalent. Les dépendances de développement ne sont pas nécessaires dans l'image de production.

La commande docker system prune peut supprimer des ressources inutilisées. Vérifiez avant de l'exécuter.

Docker dans le DevOps, CI/CD et le Cloud

Docker et DevOps

Docker est un pilier du mouvement DevOps. Il permet aux développeurs et aux ops de collaborer sur le même artefact : l'image Docker. Fini les 'ça marche chez moi', l'image qui passe les tests en CI est la même qui est déployée en production.

Docker dans les pipelines CI/CD

Dans un pipeline CI/CD typique : 1) Le développeur pousse du code sur GitHub/GitLab. 2) Le runner CI clone le code et exécute 'docker build -t app .'. 3) Les tests s'exécutent dans un conteneur temporaire. 4) L'image est taguée et pushée vers un registre. 5) L'image est déployée sur le serveur de production avec 'docker compose up'.

Outils populaires : GitHub Actions, GitLab CI/CD, Jenkins, CircleCI, AWS CodeBuild.

Docker dans le Cloud

Tous les grands fournisseurs cloud supportent Docker nativement. AWS ECS (Elastic Container Service), Azure Container Instances, Google Cloud Run. Pour l'orchestration à grande échelle, Kubernetes est le standard de facto.

AWS ECS / EKS, Azure Container Apps / AKS, Google Cloud Run / GKE, DigitalOcean App Platform.

9. Exemples Pratiques

1Exemple 1 : Lancer un serveur Web Nginx

Nginx est un serveur web extrêmement populaire. Cette commande permet d'héberger un serveur web instantanément.

bash
docker run -d -p 8080:80 nginx
  • →-d : Lance le conteneur en arrière-plan (detached mode). Le terminal reste utilisable.
  • →-p 8080:80 : Mappe le port 8080 de votre machine locale vers le port 80 (port HTTP standard) du conteneur.
  • →nginx : Spécifie le nom de l'image. Si elle n'est pas présente localement, Docker la télécharge automatiquement.
  • →Accès : Ouvrez votre navigateur sur `http://localhost:8080` pour voir la page d'accueil d'Nginx.

2Exemple 2 : Lancer une base de données MongoDB

MongoDB est une base de données NoSQL très populaire pour les applications Node.js/Express.

bash
docker run -d --name my-mongo -p 27017:27017 mongo
  • →--name my-mongo : Donne un nom convivial au conteneur afin de pouvoir l'arrêter ou le redémarrer plus facilement sans chercher son ID aléatoire.
  • →-p 27017:27017 : Ouvre le port par défaut de MongoDB pour permettre à vos outils locaux (comme Compass) de s'y connecter.

3Exemple 3 : Conteneuriser une application Node.js complète

Voyons l'ensemble des fichiers nécessaires pour faire fonctionner une application Node.js autonome sous Docker.

package.json

json
{
  "name": "docker-node-app",
  "version": "1.0.0",
  "main": "index.js",
  "scripts": {
    "start": "node index.js"
  },
  "dependencies": {
    "express": "^4.19.2"
  }
}

index.js

javascript
const express = require('express');
const app = express();
const PORT = process.env.PORT || 3000;

app.get('/', (req, res) => {
  res.send('<h1>Bonjour depuis Docker ! / Hello from Docker!</h1>');
});

app.listen(PORT, () => {
  console.log(`Serveur démarré sur le port ${PORT}`);
});

Dockerfile

dockerfile
FROM node:20-alpine
WORKDIR /usr/src/app
COPY package*.json ./
RUN npm install --only=production
COPY . .
EXPOSE 3000
CMD ["node", "index.js"]

Étape 1 : Builder l'image

bash
docker build -t node-hello-app .

→ Crée l'image nommée 'node-hello-app' en utilisant le Dockerfile présent dans le dossier actuel.

Étape 2 : Lancer le conteneur

bash
docker run -d -p 3000:3000 node-hello-app

→ Démarre l'application et la rend disponible sur `http://localhost:3000`.
